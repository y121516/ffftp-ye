# FFFTPプロジェクト指示書（AI向け）

本指示書は、Manus AI（以下「私」）がFFFTPプロジェクトにおいて開発作業を行う際の行動規範、手順、および品質基準を定めます。ユーザー様（プロジェクトオーナー）との二人三脚での開発を円滑に進めることを目的とします。

## 1. 基本原則

私は以下の原則を遵守し、開発作業を進めます。

*   **mainブランチの保護**: `main` ブランチへの直接プッシュは行いません。すべての変更はブランチ経由で行います。
*   **新規ブランチでの作業**: 作業を開始する際は、必ず `main` から新しいブランチを作成し、そのブランチ内で作業を行います。
*   **Conventional Commitsの遵守**: コミットメッセージは [Conventional Commits](https://www.conventionalcommits.org/) に従い、変更履歴の明確化と自動化に貢献します。
*   **テスト駆動開発（TDD）サイクル**: 開発はTDDの原則に従い、テストを先に書き、そのテストをパスするように実装を進めます。
*   **Single Source of Truth (SSOT) の原則**: プロジェクト内の特定のドキュメントを唯一の真実の情報源として扱い、矛盾が生じた場合はSSOTを優先します。
*   **単方向依存の原則**: 新規に作成するドキュメント、テストライブラリ、テストコードにおいては、依存関係を単方向に保ち、循環参照や不必要な結合を避けます。既存のソースコードの依存関係は現状維持としますが、新規追加・修正部分で既存の複雑な依存関係に引きずられないよう最大限注意します。

## 2. ブランチ戦略とコミット規約

### 2.1. ブランチ命名規則
作業ブランチは、その目的を明確に示すプレフィックスと、関連するタスクや機能の名前を組み合わせた形式で命名します。

*   `feat/<feature-name>`: 新機能の実装
*   `fix/<bug-description>`: バグ修正
*   `docs/<document-name>`: ドキュメントの作成・修正
*   `test/<test-scope>`: テストコードの追加・修正、テスト環境の構築
*   `chore/<task-description>`: ビルドプロセス、ツール設定、その他開発環境に関する変更
*   `refactor/<refactor-scope>`: 既存コードのリファクタリング（外部動作を変更しない改善）

### 2.2. コミットメッセージ
Conventional Commitsのタイプを適切に選択し、簡潔かつ分かりやすいメッセージを記述します。

*   例: `feat: 自動ミラーリング時の終了オプションを追加`
*   例: `docs: ゴールデンマスターテスト計画書を更新`
*   例: `test: コマンドラインオプション解析の単体テストを追加`

## 3. 開発手順（TDD/BDDベース）

機能追加、バグ修正、テスト実装は、以下のサイクルで実施します。

1.  **要件・仕様の確認（SSOT参照）**: 
    *   `ffftp_spec_doc.md` および `gmt_execution_plan.md` をSSOTとして参照し、タスクの要件と期待される動作を明確に理解します。
    *   不明点や矛盾がある場合は、ユーザー様に確認を求めます。
2.  **ドキュメント作成・修正**: 
    *   必要に応じて、`docs/plans/` または `docs/design/` 配下の関連ドキュメントを更新または新規作成します。
    *   特に、テスト計画や設計に関する変更は、この段階でドキュメントに反映させます。
3.  **テストの実装**: 
    *   要件を満たすためのテストコード（またはテストスクリプト）を `test/` ディレクトリ配下に作成します。
    *   新規テストコードは、単方向依存の原則を厳守します。
4.  **テストの失敗を確認 (Red)**: 
    *   実装前にテストを実行し、意図通りに失敗することを確認します。これにより、テストコード自体の妥当性を検証します。
5.  **実装の開始・修正 (Green)**: 
    *   テストをパスするための最小限のコードを実装または修正します。
    *   既存コードの複雑な依存関係に新規コードが引きずられないよう、インターフェースの分離などを検討します。
6.  **テストの成功を確認**: 
    *   すべてのテストがパスすることを確認します。これにより、変更が既存機能に影響を与えていないことを保証します。
7.  **リファクタリング (Refactor)**: 
    *   動作を変更せずに、コードの可読性、保守性、効率性を向上させるためのリファクタリングを行います。
8.  **Pull Requestの作成**: 
    *   変更内容をまとめ、ユーザー様にレビューを依頼します。PRには、変更の概要、テスト結果、関連するドキュメントへのリンクを含めます。

## 4. ディレクトリ構成基準

プロジェクトの主要なディレクトリ構成は以下の通りです。新規ファイルやディレクトリを作成する際は、この構造に従います。

```text
.
├── src/            # FFFTP本体のC/C++ソースコード
├── test/           # テスト関連ファイル
│   ├── golden/     # ゴールデンマスターテスト用スクリプト・データ
│   └── unit/       # 単体テストコード、テストヘルパーライブラリ
├── docs/           # ドキュメント全般
│   ├── plans/      # 開発計画書、テスト計画書、AI作業計画
│   └── design/     # 仕様書、設計ドキュメント
└── README.md       # プロジェクト概要
```

## 5. Single Source of Truth (SSOT) の定義

以下のドキュメントをSSOTとして扱います。これらのドキュメントの内容が、他の情報源と矛盾する場合は、これらのドキュメントが優先されます。

*   `docs/design/ffftp_spec_doc.md`
*   `docs/plans/gmt_execution_plan.md`

## 6. 依存関係の原則

*   **新規作成物**: 私が新規に作成するドキュメント、テストライブラリ、テストコードは、依存関係を単方向に保ちます。具体的には、上位レイヤーから下位レイヤーへの依存は許容しますが、逆方向の依存や循環依存は厳禁とします。
*   **既存コード**: 既存のFFFTPソースコードにおける複雑な依存関係は認識していますが、新規追加・修正部分がその複雑さに引きずられないよう、可能な限り疎結合な設計を心がけます。既存コードを修正する際も、依存関係の改善を意識し、リファクタリングの機会があれば積極的に提案します。
