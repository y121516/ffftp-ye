# FFFTP ゴールデンマスターテスト拡張計画書：ヘッドレス動作とログキャプチャ機構

## 1. 目的

本計画書は、FFFTPのゴールデンマスターテスト（GMT）をより詳細な動作まで拡張するため、以下の2つの主要な改善を実装するための具体的な計画を提示します。

1.  **ヘッドレス動作の追加**: GUIを介さずコマンドライン引数やスクリプトで特定のFTP操作を完結させるテスト用モードの実装。
2.  **ログキャプチャ機構**: `SetTaskMsg` の出力を標準出力またはファイルにリダイレクトする機能の実装。

これにより、自動化されたテスト環境でのFFFTPの挙動検証を効率的かつ網羅的に行えるようにします。

## 2. 現状分析

FFFTPの既存コードベースを調査した結果、以下の点が明らかになりました。

### 2.1. コマンドライン引数処理

*   `main.c` の `AnalyzeComLine` 関数がコマンドライン引数を解析し、`OPT_MIRROR`、`OPT_QUIT`、`OPT_SET` などのオプションを処理しています [^1]。
*   既存のコマンドライン操作（例: `-s <ホスト名>` での接続）は、`StartupProc` 関数内で `PostMessage` を使用してメインウィンドウ (`hWndFtp`) にメッセージを送信することで実行されます [^1]。これは、GUIが初期化されていることを前提としています。

### 2.2. GUI初期化

*   `main.c` の `WinMain` 関数から呼び出される `InitApp` 関数内で、`MakeAllWindows` がGUIの主要なウィンドウ（メインウィンドウ、タスクウィンドウなど）を初期化しています [^1]。
*   GUIの初期化は、アプリケーションの起動フローの早い段階で行われます。

### 2.3. ログ出力 (`SetTaskMsg`)

*   `taskwin.c` の `SetTaskMsg` 関数が、FFFTPのタスクウィンドウにメッセージを出力する主要なメカニズムです [^2]。
*   `SetTaskMsg` は、可変長引数を受け取り、`wvsprintf` でフォーマットした後、タスクウィンドウの `EDIT` コントロールにテキストを追加します [^2]。
*   この関数は `hWndTask` (タスクウィンドウのハンドル) がNULLでないことを前提としており、GUIが存在しない環境では機能しません。
*   `taskwin.c` には `DoPrintf2` という関数も存在し、これは `WriteConsole(GetStdHandle(STD_OUTPUT_HANDLE), ...)` を直接呼び出すことで標準出力にメッセージを書き出す機能を持っていますが、現在のところデバッグ用途に限定されており、`SetTaskMsg` の主要な出力経路ではありません [^2]。

### 2.4. ヘッドレス動作の既存の兆候

*   `connect.c` には `HostData.NoDisplayUI` というフラグが存在し、これが `YES` の場合、認証ダイアログなどのGUI要素の表示が抑制されることが確認されています [^3]。これは、GUIなしでの操作を部分的にサポートする既存のメカニズムとして活用できる可能性があります。

## 3. 実装計画

### 3.1. ヘッドレス動作モードの追加

#### 3.1.1. 新しいコマンドラインオプションの定義

*   **`common.h`**: 新しいコマンドラインオプション `OPT_HEADLESS` を定義します。
    ```c
    #define OPT_HEADLESS    0x00080000  /* ヘッドレスモードで起動 */
    ```
*   **`main.c` - `AnalyzeComLine`**: `AnalyzeComLine` 関数に `-headless` または `-hls` オプションの解析ロジックを追加し、`CmdOption` に `OPT_HEADLESS` フラグを設定するようにします [^1]。

#### 3.1.2. GUI初期化のスキップ

*   **`main.c` - `InitApp`**: `InitApp` 関数内で、`CmdOption` に `OPT_HEADLESS` フラグが設定されている場合、`MakeAllWindows(cmdShow)` の呼び出しをスキップする条件分岐を追加します [^1]。
    ```c
    // InitApp関数内
    if((CmdOption & OPT_HEADLESS) == 0) {
        if(MakeAllWindows(cmdShow) == FFFTP_SUCCESS) {
            // ... 既存のGUI関連の初期化処理 ...
        }
    } else {
        // ヘッドレスモード時の初期化処理（必要であれば）
        // 例: タスクウィンドウの代わりにログファイルや標準出力へのハンドルを設定
    }
    ```

#### 3.1.3. GUI依存機能の代替とリファクタリング

*   **`main.c` - `StartupProc`**: `StartupProc` 関数内で、`OPT_HEADLESS` フラグが設定されている場合、`PostMessage(hWndFtp, WM_COMMAND, ...)` を使用する代わりに、対応するFTP操作を直接実行する内部関数を呼び出すようにロジックを変更します [^1]。これにより、GUIを介さずに操作が実行されます。
    *   例えば、`MENU_CONNECT_NUM` に対応する処理を直接呼び出す関数 (`ConnectToHost(AutoConnect, CmdOption)`) を新たに作成し、`StartupProc` から呼び出すことを検討します。
*   **`connect.c`**: `HostData.NoDisplayUI` フラグの活用を拡張し、ヘッドレスモード時には常に `YES` に設定されるようにします。これにより、接続時の認証ダイアログなどが表示されなくなります [^3]。

### 3.2. ログキャプチャ機構の実装

#### 3.2.1. 新しいコマンドラインオプションの定義

*   **`common.h`**: 新しいコマンドラインオプション `OPT_LOGFILE` を定義します。
    ```c
    #define OPT_LOGFILE     0x00100000  /* ログファイルを指定 */
    ```
*   **`main.c` - `AnalyzeComLine`**: `AnalyzeComLine` 関数に `-logfile <path>` オプションの解析ロジックを追加し、指定されたログファイルパスをグローバル変数に保存するようにします [^1]。

#### 3.2.2. `SetTaskMsg` のリファクタリング

*   **`taskwin.c` - `SetTaskMsg`**: `SetTaskMsg` 関数の内部ロジックを以下のように変更します [^2]。
    1.  グローバルなログ出力モードフラグ（例: `LogOutputMode`）とログファイルハンドル（例: `hLogFile`）を導入します。
    2.  `OPT_HEADLESS` が設定されている場合、`LogOutputMode` を「標準出力」または「ファイル」に設定します。
    3.  `OPT_LOGFILE` が指定されている場合、`hLogFile` をオープンし、`LogOutputMode` を「ファイル」に設定します。
    4.  `SetTaskMsg` の中で、`LogOutputMode` に応じて出力先を切り替えます。
        *   **GUIモード**: 既存の `EDIT` コントロールへの書き込み (`SendMessage(GetTaskWnd(), ...)`)
        *   **標準出力モード**: `WriteConsole(GetStdHandle(STD_OUTPUT_HANDLE), ...)` を使用して標準出力に書き込みます。
        *   **ファイルモード**: `fprintf(hLogFile, ...)` を使用して指定されたログファイルに書き込みます。
*   **`taskwin.c` - `DoPrintf`**: `DoPrintf` は現在 `SetTaskMsg` を呼び出しているため、`SetTaskMsg` の変更により自動的にログキャプチャの恩恵を受けます [^2]。
*   **`main.c` - 終了処理**: アプリケーション終了時に、開いているログファイル (`hLogFile`) を確実にクローズする処理を追加します。

### 3.3. 開発プロセス

1.  **ブランチ作成**: 新しい機能開発用のブランチを作成します（例: `feature/headless-gmt`）。
2.  **段階的な実装**: 上記計画に従い、ヘッドレスモードのオプション解析、GUI初期化スキップ、`SetTaskMsg` のリファクタリング、`StartupProc` の変更を段階的に実装します。
3.  **単体テスト**: 各変更が意図通りに動作するか、単体テストを実施します。
4.  **統合テスト**: ヘッドレスモードでFTP操作を実行し、ログが正しく出力されることを確認します。
5.  **コードレビュー**: 実装完了後、コードレビューを実施し、品質と安定性を確保します。
6.  **マージ**: レビューとテストが完了した後、メインブランチにマージします。

## 4. 影響範囲

*   **既存のGUI動作**: ヘッドレスモードが指定されない限り、既存のGUI動作に影響を与えないように慎重に実装します。
*   **パフォーマンス**: ログ出力のリダイレクトは、大量のログが出力される場合にパフォーマンスに影響を与える可能性があります。必要に応じてバッファリングなどの最適化を検討します。
*   **エラーハンドリング**: ヘッドレスモードでのエラー発生時も、ログに詳細な情報が出力されるようにエラーハンドリングを強化します。

## 5. 今後の展望

本計画の実装後、以下の機能拡張を検討します。

*   **JSON形式ログ出力**: 機械可読性の高いJSON形式でのログ出力をサポートし、テスト結果の解析を容易にする。
*   **設定ファイルからの操作**: コマンドライン引数だけでなく、設定ファイル（例: YAML, JSON）から複雑なFTP操作シーケンスを定義・実行できるようにする。
*   **詳細なテストレポート**: ログデータから自動的にテスト結果レポートを生成するツールを開発する。
