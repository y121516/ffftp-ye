# FFFTP ゴールデンマスターテスト実施計画書

## 1. 目的

本計画書の目的は、FFFTPのソースコード近代化および機能追加プロジェクトにおいて、**FFFTP 2.0の通信挙動、GUI操作感、コマンドライン動作の完全な互換性を担保する**ためのゴールデンマスターテストを確立することです。特に、ソースコードに一切変更を加えない「非侵襲的アプローチ」を採用し、テスト基盤の導入自体による「2.0の挙動」への影響を排除します。

## 2. スコープ

本ゴールデンマスターテストのスコープは、FFFTPの主要な機能におけるプロトコルレベルの挙動およびユーザーインターフェースの動作確認に焦点を当てます。

*   **FTP/FTPS/SFTP接続**: 各種プロトコルでの接続確立、認証、切断のシーケンス。
*   **ファイル転送**: アップロード、ダウンロード、ミラーリング（アップロード/ダウンロード）におけるデータ転送の正確性、速度、エラー処理。
*   **文字コード変換**: ファイル内容およびファイル名の文字コード（SJIS, EUC, JIS, UTF-8）変換の正確性。
*   **特殊ファイル名**: 日本語、スペース、特殊記号を含むファイル名の処理。
*   **PASV/PORTモード**: データコネクション確立方式の切り替えと挙動。
*   **GUI操作**: ホスト接続、ファイル操作、設定変更などのGUIを介した操作が、コマンドライン操作と同様のプロトコル挙動を生成すること。
*   **コマンドラインからの自動操作**: コマンドライン引数とINIファイルによるFFFTPの自動起動、操作、終了。

## 3. 前提条件

ゴールデンマスターテストを実施するために、以下の環境とリソースが必要です。

*   **FFFTP 2.0 オリジナルバイナリ**: 互換性の基準となる、変更が加えられていないFFFTP 2.0の実行ファイル (`ffftp.exe`)。
*   **開発中FFFTPバイナリ**: 近代化作業によってビルドされたFFFTPの実行ファイル。
*   **テスト用FTPサーバー**: 挙動が固定されたローカルFTPサーバー（例: Dockerコンテナで動作するvsftpd, ProFTPD）。サーバー側の設定（文字コード、ファイルシステムなど）も固定し、テストデータが配置されていること。
*   **テスト用データセット**: 各テストシナリオに対応するローカル/リモートのファイル群。これには、様々な文字コード（SJIS, EUC, UTF-8）、ファイルサイズ（0バイト、小、大）、ファイルタイプ（テキスト、バイナリ）、特殊文字を含むファイル名などが含まれます。
*   **スクリプト実行環境**: テストの自動化を担うPowerShellまたはPythonの実行環境。
*   **テキスト比較ツール**: ログファイルの差異を検出するためのツール（例: `diff`コマンド、WinMergeなど）。

## 4. 実施手順

本ゴールデンマスターテストは、以下のステップで実施されます。

### 4.1. ステップ1: テスト環境の準備

1.  **FFFTPバイナリの配置**: FFFTP 2.0オリジナルバイナリと開発中バイナリをそれぞれ異なるディレクトリに配置します。
2.  **テスト用FTPサーバーのセットアップ**: Docker等を用いて、挙動が固定されたFTPサーバーをセットアップします。サーバーのルートディレクトリには、テストに必要なファイル（アップロード元、ダウンロード先、ミラーリング対象など）を配置します。
3.  **テスト用データセットの準備**: 各シナリオで利用するファイル（日本語ファイル名、特殊記号ファイル名、大容量ファイルなど）をローカルおよびリモートのテストディレクトリに配置します。

### 4.2. ステップ2: テストシナリオの定義とINIファイルの生成

1.  **テストシナリオのリストアップ**: 「FFFTP 仕様ドキュメント」の「3. テストシナリオと設定項目のマッピング」セクションに基づき、具体的なテストシナリオを詳細化します。各シナリオは、以下の要素を明確にします。
    *   **目的**: 何をテストするのか。
    *   **操作**: 接続、アップロード、ダウンロード、ミラーリングなど。
    *   **期待される結果**: 成功/失敗、特定のプロトコルコマンドの送受信、ファイル内容の一致など。
2.  **INIファイル生成スクリプトの作成**: 各テストシナリオに対応する `ffftp.ini` ファイルを動的に生成するスクリプト（例: Pythonスクリプト）を作成します。このスクリプトは、以下の設定をINIファイルに含めます。
    *   `[FFFTP\Options]` セクション: `Debug=1` (デバッグコンソール有効化)、`WinPos=0` (ウィンドウ位置保存無効化)。
    *   `[FFFTP\Options\HostN]` セクション: 接続先情報 (`HostName`, `HostAdrs`, `UserName`, `Password`, `Port`)、文字コード設定 (`Kanji`, `NameKanji`)、転送モード (`TrType`)、PASVモード (`Pasv`)、暗号化プロトコル (`FTPES`, `FTPIS`, `SFTP`)、エラーモード (`ErrMode`)、ウィンドウ位置 (`WinPos`) など、シナリオに必要な設定。

### 4.3. ステップ3: ゴールデンマスターログの取得（FFFTP 2.0）

1.  **FFFTP 2.0の実行**: 各テストシナリオに対して、FFFTP 2.0オリジナルバイナリをコマンドラインから実行します。INIファイルはステップ2で生成したものを使用し、`-f` (強制実行)、`-q` (自動終了) オプションを付与します。
    ```bash
    ffftp.exe -n 
"C:\test\scenario1.ini" -s "TestHost" -m -f -q
    ```
2.  **ログの回収**: FFFTPの実行が完了した後、INIファイルで指定されたログファイル（例: `ffftp.log`）を回収します。
3.  **ログの正規化**: 回収したログファイルから、実行ごとに変動する要素（タイムスタンプ、ソケットID、プロセスID、一時ファイルパスなど）を正規表現を用いて一貫したプレースホルダーに置換します。この処理は、PythonやPowerShellスクリプトで自動化します。
    *   **タイムスタンプ**: `YYYY/MM/DD HH:MM:SS` -> `[TIMESTAMP]`
    *   **ソケットID**: `(S=0xXXXXXXXX)` -> `(S=[SOCKET_ID])`
    *   **プロセスID**: `PID: XXXXX` -> `PID: [PID]`
    *   **一時ファイルパス**: `C:\Users\...\temp\` -> `[TEMP_PATH]`
4.  **ゴールデンマスターログの保存**: 正規化されたログファイルを、各シナリオに対応する「ゴールデンマスターログ」としてバージョン管理システムに保存します。例: `golden_master/scenario1_upload_utf8.log`

### 4.4. ステップ4: 開発中FFFTPのテスト実行と結果比較

1.  **開発中FFFTPの実行**: ステップ4.3.1と全く同じコマンドライン引数とINIファイルを用いて、開発中のFFFTPバイナリを実行します。
2.  **ログの回収と正規化**: ステップ4.3.2および4.3.3と同様に、出力されたログファイルを回収し、正規化します。
3.  **ログの比較**: 正規化された開発中FFFTPのログと、対応するゴールデンマスターログをテキスト比較ツール（`diff`コマンドなど）で比較します。差異がないことを確認します。
    *   **差異がない場合**: テストは成功です。FFFTP 2.0との互換性が維持されています。
    *   **差異がある場合**: テストは失敗です。差異の内容を分析し、FFFTP 2.0の挙動からの逸脱が意図的なものか、バグによるものかを判断します。意図的な変更でない場合は、修正が必要です。

## 5. テスト自動化ツールとスクリプト

本計画では、以下のツールとスクリプトを活用してテストプロセスを自動化します。

*   **INIファイル生成スクリプト**: PythonまたはPowerShellで実装し、テストシナリオ定義に基づいてINIファイルを動的に生成します。
*   **FFFTP実行スクリプト**: PowerShellの `Start-Process` コマンドなどを利用し、FFFTPの起動、INIファイルの指定、コマンドライン引数の渡し、終了コードの取得を行います。
*   **ログ正規化スクリプト**: Pythonの正規表現ライブラリ (`re`) やPowerShellの `Select-String`、`Replace` などを利用し、ログから変動要素を除去します。
*   **ログ比較スクリプト**: `diff` コマンド（Linux環境）や `Compare-Object` (PowerShell) を利用し、正規化されたログファイルを比較します。
*   **テストランナー**: これらのスクリプトを統合し、一連のテストシナリオを自動実行する上位スクリプト（例: Pythonの `unittest` フレームワークやPowerShellスクリプト）を構築します。

## 6. 継続的インテグレーション (CI) との連携

本ゴールデンマスターテストは、GitHub ActionsなどのCI/CDパイプラインに組み込むことで、コード変更ごとに自動的に互換性検証を実行できます。これにより、FFFTP 2.0の挙動からの意図しない逸脱を早期に検出し、品質を継続的に保証することが可能となります。

## 7. 今後の課題

*   **GUI操作のテスト**: コマンドラインやINIファイルで制御できないGUI操作（例: ドラッグ＆ドロップ、右クリックメニュー）の挙動をゴールデンマスターテストに含めるには、UI自動化ツール（例: WinAppDriver, Selenium for Desktop）の導入検討が必要です。ただし、これはソースコードへの変更を伴う可能性があり、非侵襲的アプローチの原則とのバランスを考慮する必要があります。
*   **パフォーマンス測定**: 転送速度やCPU/メモリ使用量などのパフォーマンス指標も、将来的にゴールデンマスターテストの対象として追加を検討します。
